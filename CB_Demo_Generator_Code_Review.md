# Code Review: cb_demo_generator.jsx

> **[SUPERSEDED — 2026-04-25]** This review targets `cb_demo_generator.jsx`, which has been moved to `archive/` and is no longer part of the active codebase. The architecture has since pivoted to a Flask + React local server (`server.py` + `cb_demo.html`), which resolved Issues #1, #2, #3, #4, and #5 by design. This document is preserved for historical context only. For the current architecture, see `README.md`.

## Summary

A clever single-file React artifact that takes a domain template, calls Claude to generate a 4-tier Codebeamer stream hierarchy as JSON, then compiles it into a downloadable Python provisioning script. The architecture concept is sound and the UI is polished. However, there are **2 critical issues** (the Anthropic API call will fail as written, and credentials are embedded in plaintext), **3 high-severity correctness problems** (token limit too low, stream hierarchy not automated, traceability method questionable), and several maintainability improvements that would make this production-ready.

## Critical Issues

| # | Lines | Issue | Severity |
|---|-------|-------|----------|
| 1 | 491–499 | **Anthropic API call has no `x-api-key` header** — the `fetch` to `api.anthropic.com/v1/messages` only sends `Content-Type`. This will return 401 every time. Either the API key needs to be provided (with major security implications if client-side), or the call needs to go through a backend proxy. | 🔴 Critical |
| 2 | 183–184 | **Credentials embedded as plaintext in generated Python** — `USER` and `PASS` are written directly into the script source. Anyone who receives the `.py` file gets the Codebeamer credentials. Should use environment variables or a config file. | 🔴 Critical |

### Fix for Issue #1

The component currently can't reach the Anthropic API. There are two paths:

**Option A — If this runs inside Claude's artifact environment**, the AI generation should be handled differently (the artifact can't make external API calls). The JSON structure should be pre-generated by Claude and passed as props, or the component should accept the structure as user-pasted JSON.

**Option B — If this is a standalone app**, add API key handling:

```jsx
// In the CB Config step, add an API key field
const [apiKey, setApiKey] = useState("");

// In the fetch call
headers: {
  "Content-Type": "application/json",
  "x-api-key": apiKey,
  "anthropic-version": "2023-06-01",  // also missing!
  "anthropic-dangerous-direct-browser-access": "true"
},
```

But **Option B is not recommended** — API keys should never live in browser-side code. The proper architecture is a thin backend proxy.

Additionally, the prompt asks for very specific quantities but the token budget can't deliver. Consider generating in two passes (structure first, then details) or reducing the per-item field count.

### Fix for Issue #4

The Python script should establish the stream hierarchy. Modify `buildPythonScript` to track stream IDs per tier and set `sourceStreamId` when creating child streams:

```python
def mk_stream(name, desc="", source_stream_id=None):
    body = {"name": name, "color": STREAM_COLOR, "description": desc}
    if source_stream_id:
        body["sourceStreamId"] = source_stream_id
        body["isSourceStreamChecked"] = True
    r = _post("/v3/streams/stream", body)
    return r["id"]
```

Then in the generated code, pass library stream IDs into PL stream creation, PL IDs into transform creation, etc.

### Fix for Issue #5

Replace `set_verifies` with a proper association:

```python
def mk_association(from_id, to_id, type_id=2):
    """POST /v3/associations — type 2 is typically 'depends on' / 'verifies'
    Check GET /v3/associations/types for your instance's type IDs"""
    if not from_id or not to_id:
        return
    try:
        _post("/v3/associations", {
            "from": {"id": from_id},
            "to": {"id": to_id},
            "type": {"id": type_id}
        })
        ok(f"    [ASSOC] {from_id} --verifies--> {to_id}")
    except Exception as e:
        warn(f"    [ASSOC] {from_id}->{to_id}: {e}")
```

### Fix for Issue #8 — Retry wrapper

```python
import time

def _request(method, path, body=None, retries=3):
    for attempt in range(retries):
        try:
            if method == "GET":
                r = requests.get(f"{BASE}{path}", auth=AUTH, headers=HEADERS)
            elif method == "POST":
                r = requests.post(f"{BASE}{path}", auth=AUTH, headers=HEADERS, json=body)
            elif method == "PUT":
                r = requests.put(f"{BASE}{path}", auth=AUTH, headers=HEADERS, json=body)
            if r.status_code == 429:
                wait = min(2 ** attempt, 10)
                warn(f"Rate limited, waiting {wait}s...")
                time.sleep(wait)
                continue
            r.raise_for_status()
            return r.json()
        except requests.exceptions.ConnectionError:
            if attempt < retries - 1:
                time.sleep(1)
                continue
            raise
    raise Exception(f"Failed after {retries} retries: {method} {path}")
```

## What Looks Good

- **The 4-tier architecture mapping is well-conceived.** The TIERS config, color scheme, and tier-to-stream mapping directly reflects the ADR we produced. This is a solid implementation of the strategy.
- **The AI prompt (lines 37–108) is excellent.** Very specific constraints, real standards references (IEC 60601, ISO 26262, DO-178C), shall-style requirements with measurable values — this will generate convincing demo data.
- **The JSON sanitizer (lines 113–154) is thoughtful.** State-machine-based string escape handling with three-level fallback is robust defensive programming.
- **UI/UX is clean and professional.** The stepped wizard, domain template picker, stream tree preview with expand/collapse, and dual-tab output (preview vs. script) are well-designed.
- **Generated Python is well-documented.** Each function has a docstring referencing the exact API endpoint, method, and schema requirements.

---

## Verdict

**Request Changes** — The two critical issues (no API key = nothing works, plaintext credentials in output) must be fixed before this can be used. The `max_tokens` and stream hierarchy issues should be addressed in the same pass since they directly impact demo quality.

### Priority order for fixes:
1. Resolve the API authentication strategy (Issues #1 and #10)
2. Remove plaintext credentials from generated script (Issue #2)
3. Increase max_tokens to 4096 (Issue #3)
4. Add sourceStreamId to establish stream hierarchy (Issue #4)
5. Switch to POST /v3/associations for traceability (Issue #5)
6. Add retry logic with rate-limit handling (Issue #8)
